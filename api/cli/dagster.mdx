# dagster CLI

The `dagster` command-line interface provides tools for working with Dagster projects, including job execution, asset materialization, schedule management, and development workflows.

## Installation

```bash
pip install dagster
```

## Global Options

<ParamField path="--version" type="flag">
  Show the installed Dagster version
</ParamField>

<ParamField path="--help" type="flag">
  Show help message and exit
</ParamField>

## dagster dev

Start a local development deployment of Dagster, including the webserver and daemon.

<Info>
  **Superseded**: This command is being replaced by `dg dev`. The `dagster dev` command will continue to work but consider migrating to the new CLI.
</Info>

### Usage

```bash
dagster dev [OPTIONS]
```

### Options

<ParamField path="--code-server-log-level" type="choice" default="warning">
  Set the log level for code servers spun up by dagster services
  
  **Choices**: `critical`, `error`, `warning`, `info`, `debug`
</ParamField>

<ParamField path="--log-level" type="choice" default="info">
  Set the log level for dagster services
  
  **Choices**: `critical`, `error`, `warning`, `info`, `debug`
</ParamField>

<ParamField path="--log-format" type="choice" default="colored">
  Format of the logs for dagster services
  
  **Choices**: `colored`, `json`, `rich`
</ParamField>

<ParamField path="--port" type="integer">
  Port to use for the Dagster webserver
  
  **Aliases**: `--dagit-port`, `-p`
</ParamField>

<ParamField path="--host" type="string">
  Host to use for the Dagster webserver
  
  **Aliases**: `--dagit-host`, `-h`
</ParamField>

<ParamField path="--live-data-poll-rate" type="string" default="2000">
  Rate at which the dagster UI polls for updated asset data (in milliseconds)
</ParamField>

<ParamField path="--use-legacy-code-server-behavior" type="flag">
  Use the legacy behavior of the daemon and webserver each starting up their own code server
</ParamField>

<ParamField path="-v, --verbose" type="flag">
  Show verbose stack traces for errors in the code server
</ParamField>

### Workspace Options

<ParamField path="-w, --workspace" type="path">
  Path to workspace file. Argument can be provided multiple times
</ParamField>

<ParamField path="-f, --python-file" type="path">
  Path to a Python file containing Dagster definitions
</ParamField>

<ParamField path="-m, --module-name" type="string">
  Python module containing Dagster definitions
</ParamField>

<ParamField path="-d, --working-directory" type="path">
  Working directory to use when loading the repository or job
</ParamField>

<ParamField path="-a, --attribute" type="string">
  Attribute within the module or file that defines the Definitions object
</ParamField>

### Example

<CodeGroup>
```bash Basic Usage
# Start development server with default settings
dagster dev
```

```bash Custom Port
# Start on a specific port
dagster dev --port 3001
```

```bash With Python File
# Point to a specific Python file
dagster dev -f my_project/definitions.py
```

```bash With Module
# Load from a Python module
dagster dev -m my_project.definitions
```

```bash Debug Mode
# Enable verbose logging
dagster dev --log-level debug --verbose
```
</CodeGroup>

### Output

```
Launching Dagster services...

Serving dagster-webserver on http://127.0.0.1:3000 in process 12345
Starting dagster-daemon in process 12346
```

### Environment Variables

- `DAGSTER_HOME`: Directory for Dagster instance configuration (dagster.yaml)
- `DAGSTER_IS_DEV_CLI`: Automatically set to "1" when using dagster dev

## dagster job

Commands for working with Dagster jobs.

### dagster job list

List all jobs in a repository.

```bash
dagster job list [OPTIONS]
```

#### Example

```bash
dagster job list -f my_jobs.py
```

#### Output

```
Repository my_repo
******************
Job: daily_sales_job
Description:
    Process daily sales data and update reports
Ops: (Execution Order)
    load_data
    transform_data
    generate_report
```

### dagster job print

Print detailed information about a job.

```bash
dagster job print [OPTIONS]
```

#### Options

<ParamField path="--verbose" type="flag">
  Show detailed op information including inputs and outputs
</ParamField>

<ParamField path="-j, --job" type="string">
  Name of the job to print
</ParamField>

#### Example

<CodeGroup>
```bash Basic
dagster job print -f my_jobs.py -j my_job
```

```bash Verbose
dagster job print -f my_jobs.py -j my_job --verbose
```
</CodeGroup>

### dagster job execute

Execute a job in the same Python environment.

<Info>
  **Superseded**: Use `dg launch --job <job_name>` instead for new projects.
</Info>

```bash
dagster job execute [OPTIONS]
```

#### Options

<ParamField path="-j, --job" type="string">
  Name of the job to execute
</ParamField>

<ParamField path="-c, --config" type="path">
  YAML or JSON file with run configuration. Can be specified multiple times
</ParamField>

<ParamField path="--tags" type="json">
  JSON string of tags to use for this job run
  
  **Example**: `'{"team": "data", "priority": "high"}'`
</ParamField>

<ParamField path="-o, --op-selection" type="string">
  Specify op subselection to execute
  
  **Examples**:
  - `some_op` - Execute only this op
  - `*some_op` - Execute this op and all ancestors
  - `*some_op+++` - Execute this op, ancestors, and descendants within 3 levels
  - `*some_op,other_op+` - Multiple selections
</ParamField>

#### Example

<CodeGroup>
```bash Basic Execution
dagster job execute -f jobs.py -j my_job
```

```bash With Config
dagster job execute -f jobs.py -j my_job -c config.yaml
```

```bash With Tags
dagster job execute \
  -f jobs.py \
  -j my_job \
  --tags '{"owner": "data-team"}'
```

```bash Op Selection
# Execute only specific ops
dagster job execute -f jobs.py -j my_job -o "*transform_data"
```
</CodeGroup>

### dagster job launch

Launch a job using the configured run launcher.

```bash
dagster job launch [OPTIONS]
```

#### Options

<ParamField path="-j, --job" type="string">
  Name of the job to launch
</ParamField>

<ParamField path="-c, --config" type="path">
  YAML or JSON file with run configuration
</ParamField>

<ParamField path="--config-json" type="json">
  JSON string of run config (cannot be used with -c/--config)
</ParamField>

<ParamField path="--tags" type="json">
  JSON string of tags for the run
</ParamField>

<ParamField path="--run-id" type="string">
  Custom ID for the launched run
</ParamField>

<ParamField path="-o, --op-selection" type="string">
  Op subselection to execute
</ParamField>

#### Example

```bash
dagster job launch -w workspace.yaml -j my_job -c config.yaml
```

### dagster job scaffold_config


#### Options

<ParamField path="--print-only-required" type="flag">
  Only include required configuration fields

#### Example

<CodeGroup>

### dagster job backfill

Backfill a partitioned job.

```bash
dagster job backfill [OPTIONS]
```

#### Options

<ParamField path="--partitions" type="string">
  Comma-separated list of partition names to backfill
</ParamField>

<ParamField path="--all" type="flag">
  Select all partitions to backfill
</ParamField>

<ParamField path="--from" type="string">
  Start partition for the backfill range
</ParamField>

<ParamField path="--to" type="string">
  End partition for the backfill range
</ParamField>

<ParamField path="--tags" type="json">
  JSON string of tags for the backfill runs
</ParamField>

<ParamField path="--noprompt" type="flag">
  Skip confirmation prompt
</ParamField>

#### Example

<CodeGroup>
```bash Date Range
dagster job backfill \
  -w workspace.yaml \
  -j daily_partitioned_job \
  --from 2024-01-01 \
  --to 2024-01-31
```

```bash Specific Partitions
dagster job backfill \
  -w workspace.yaml \
  -j my_job \
  --partitions "2024-01-01,2024-01-15,2024-02-01"
```

```bash All Partitions
dagster job backfill -w workspace.yaml -j my_job --all --noprompt
```
</CodeGroup>

## dagster asset

Commands for working with Dagster assets.

### dagster asset materialize

Execute a run to materialize a selection of assets.

<Info>
  **Superseded**: Use `dg launch --assets <selection>` instead.
</Info>

```bash
dagster asset materialize [OPTIONS]
```

#### Options

<ParamField path="--select" type="string" required>
  Comma-separated asset selection to target
  
  **Examples**:
  - `my_asset` - Single asset
  - `my_asset*` - Asset and all downstream dependencies
  - `*my_asset` - Asset and all upstream dependencies
  - `tag:team=data` - All assets with tag
</ParamField>

<ParamField path="--partition" type="string">
  Asset partition to target
</ParamField>

<ParamField path="--partition-range" type="string">
  Asset partition range in format `<start>...<end>`. Requires all assets to have `BackfillPolicy.single_run()` policy
  
  **Example**: `2025-01-01...2025-01-05`
</ParamField>

<ParamField path="-c, --config" type="path">
  YAML or JSON file with run configuration
</ParamField>

<ParamField path="--config-json" type="json">
  JSON string of run config
</ParamField>

#### Example

<CodeGroup>
```bash Single Asset
dagster asset materialize --select my_asset -f definitions.py
```

```bash Multiple Assets
dagster asset materialize --select "asset_a,asset_b,asset_c" -f definitions.py
```

```bash With Dependencies
dagster asset materialize --select "my_asset*" -f definitions.py
```

```bash Partitioned Asset
dagster asset materialize \
  --select daily_metrics \
  --partition 2024-01-15 \
  -f definitions.py
```

```bash Partition Range
dagster asset materialize \
  --select daily_metrics \
  --partition-range "2024-01-01...2024-01-31" \
  -f definitions.py
```
</CodeGroup>

## Troubleshooting

### Command Not Found

If you get a "command not found" error:

1. Verify Dagster is installed:
   ```bash
   pip list | grep dagster
   ```

2. Check your PATH includes the Python scripts directory:
   ```bash
   which dagster
   ```

3. Try running with the Python module syntax:
   ```bash
   python -m dagster --help
   ```

### Configuration Issues

**Check DAGSTER_HOME:**
```bash
echo $DAGSTER_HOME
```

If not set, Dagster will use an ephemeral instance.

**Verify instance configuration:**
```bash
dagster instance info
```

### Workspace Loading Errors

If you encounter workspace loading errors:

1. Verify your workspace.yaml is valid
2. Check that Python modules are importable:
   ```bash
   python -c "import my_module"
   ```
3. Ensure your working directory is correct when using relative paths

### Permission Errors

If you get permission errors with the instance:

1. Check DAGSTER_HOME permissions:
   ```bash
   ls -la $DAGSTER_HOME
   ```

2. Ensure the dagster.yaml file is readable:
   ```bash
   cat $DAGSTER_HOME/dagster.yaml
   ```
